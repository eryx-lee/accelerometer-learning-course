Module 6 Cleaning and Standardizing

GGIR processing creates many output files. Cleaning turns those outputs into a documented set of variables that can be merged, filtered, and analysed consistently.

By the end of this module, you should be able to:

  1. Identify the main GGIR output files used for day, night, and person-level cleaning.
  2. Decode common GGIR part 5 file names.
  3. Apply sleep-cleaning rules and understand why MVPA bouts may need to be generated outside GGIR.
  4. Build a data dictionary that links variables back to source files.

6.1 Lesson 6.1: Know the real GGIR output files

The exact output names can vary by GGIR version and options. In the course workflow, learners should recognize these common files:

Output file pattern Unit of observation Typical use
part2_summary Participant/file Overall recording, calibration, and data-quality summaries.
part2_daysummary Participant-day Day-level wear, activity, and quality information.
part4_nightsummary Participant-night Sleep/night summaries used for sleep cleaning.
part5_personsummary_WW_..._T... Participant-level summary Person-level outcomes for selected time windows and thresholds.

Before cleaning, confirm the unit of observation. A common mistake is merging a day-level file to a person-level file without preserving day identifiers.

示例: Pelotas/UFPel cohort
Treat GGIR outputs as structured evidence: keep part 2, part 4, and part 5 files in their original output folders; copy only the files needed for cleaning into the Stata working folder. Keep a record of which lot or processing batch each file came from.

6.2 Lesson 6.2: Decode GGIR file names

GGIR file names often encode the cut-points and thresholds used. For example:

part5_personsummary_WW_L30M200V700_T5A3.csv

Read this as:

Segment Meaning
part5_personsummary Participant-level summary from part 5.
WW Time window or day-window summary label.
L30 Light-intensity threshold/cut-point = 30.
M200 Moderate-intensity threshold/cut-point = 200.
V700 Vigorous-intensity threshold/cut-point = 700.
T5 timethreshold = 5.
A3 anglethreshold = 3.

The values in a file name are not decoration; they are part of the analysis method. If two files differ only in L/M/V, T, or A, they are different processing outputs.

6.3 Lesson 6.3: Sleep cleaning

Sleep outputs often include codes that need to be reviewed before analysis. In the course example, sleep values coded 2–6 are dropped during cleaning. The exact meaning of each code should be documented from the study's GGIR output/data dictionary before exclusion.

Minimum sleep-cleaning steps:

  1. Import part4_nightsummary.
  2. Confirm participant ID and night/date variables.
  3. Review the sleep-status or quality code used by the study.
  4. Drop values coded 2–6 when this is the approved Pelotas/UFPel rule.
  5. Save a cleaned night-level file and document how many records were removed.

Study-specific rule: Dropping sleep values 2–6 is an example rule from the course pipeline. Your study should confirm which codes represent unusable or non-target sleep records before applying the same filter.

6.4 Lesson 6.4: MVPA bouts not automatically produced by GGIR

GGIR can produce many activity summaries, but some bout variables may need to be generated in Stata or another downstream tool. In this course example, the Stata cleaning workflow creates:

  • 1-minute MVPA bouts,
  • 5-minute MVPA bouts,
  • 10-minute MVPA bouts,
  • 20-minute inactivity variables.

These variables require a clear definition of what counts as a bout, how interruptions are handled, and whether the rule uses raw acceleration metrics, epoch counts, or derived intensity classifications.

6.5 Lesson 6.5: Standardize variable names

Build a data dictionary that states:

  1. original variable name,
  2. standardized variable name,
  3. source file,
  4. unit of observation,
  5. unit of measurement,
  6. transformation,
  7. missing-value rule,
  8. reason for exclusion or retention.

6.6 Wrap-up

Cleaning should preserve traceability. Every standardized variable should point back to a source file and a processing decision.
